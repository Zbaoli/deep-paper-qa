## 四、问题发现

### 致命

无。

### 重要

**1. 多数 research 题目只调用 1 次 ask_user（仅计划确认）**

6 道通过的题中，只有 #33 和 #35 完成了完整的多阶段交互（4 次 ask_user）。其余 4 道只在计划确认阶段调用 1 次 ask_user，之后一口气完成所有检索并直接输出最终回答，跳过了"每个子问题完成后调用 ask_user"的要求。

**影响**: 用户无法在研究过程中介入引导方向，Deep Research 的核心交互设计未被严格执行。

**2. #34 完全未进入工具调用阶段**

Agent 生成了研究计划但未调用任何 tool，将计划当作最终回答输出。

**影响**: 用户触发 /research 后得到的只是一个"计划"而不是研究结果。

### 改进

**3. 评测环境数据库不可用导致工具全部报错**

所有检索工具因 `'Settings' object has no attribute 'database_url'` 报错。这影响了 Agent 的行为（如 #34 可能因为看到连续报错而放弃调用工具）。

**影响**: 评测结果不能完全反映真实环境下的表现。需要修复 config 兼容性或提供评测专用 mock 数据。

## 五、优化建议

**建议 1：强化 prompt 中 ask_user 的调用要求**

问题描述：Agent 经常跳过子问题间的 ask_user 调用
影响范围：6/7 的 research 题目

```diff
--- a/src/deep_paper_qa/prompts.py
+++ b/src/deep_paper_qa/prompts.py
 第二步：逐个执行子问题
 - 按计划顺序执行每个子问题，每个子问题最多调用 2 次检索工具
 - 每个子问题完成后，调用 ask_user() 展示该阶段发现的关键论文和结论
+- 重要：每完成一个子问题后必须调用 ask_user()，禁止连续执行多个子问题后才调用
+- 禁止在未调用 ask_user() 的情况下直接进入下一个子问题或生成最终总结
```

预期：Agent 在每个子问题完成后严格暂停等待用户指令

**建议 2：强制计划阶段必须调用 ask_user**

问题描述：#34 Agent 输出计划后直接结束，未调用 ask_user
影响范围：Q34

```diff
--- a/src/deep_paper_qa/prompts.py
+++ b/src/deep_paper_qa/prompts.py
 第一步：制定研究计划
 - 分析用户问题，分解为 3-5 个可检索的子问题
 - 为每个子问题注明计划使用的工具（execute_sql / search_abstracts / vector_search）
 - 调用 ask_user() 展示研究计划，等待用户确认或调整
+- 重要：研究计划必须通过 ask_user() 工具展示，不要直接输出计划文本作为回答
+- 制定计划后的第一个动作必须是调用 ask_user()，不允许跳过
```

预期：Agent 始终通过 ask_user() 展示计划并等待确认

**建议 3：修复评测环境 config 兼容性**

问题描述：worktree 的 config.py 使用 `pg_database_url` 但旧版 execute_sql 引用 `database_url`
影响范围：所有评测题目的实际数据库查询

```diff
--- a/eval/run_eval.py
+++ b/eval/run_eval.py
+# 评测时也 mock 数据库工具，返回预设数据
+# 或确保 .env 中的配置项与 config.py 的字段名一致
```

预期：评测能在无真实数据库环境下可靠运行，或 config 字段名统一后连接真实数据库
