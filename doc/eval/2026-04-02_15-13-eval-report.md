## 四、问题发现

### 重要：mixed 题路由不稳定（76.9%）

**影响范围**：Q23、Q24、Q34
**根因**：prompt 中工具选择规则写了"统计/计数/排名/趋势 → execute_sql"，但这 3 题的意图（"代表性工作有哪些进展"、"论文都用了哪些数据集"、"主要解决什么问题"）不包含这些关键词，agent 将其识别为纯搜索问题。
**严重程度**：中等 — 回答质量不一定差，但缺少 SQL 统计维度。

### 改进：content 题工具调用仍偏多

**影响范围**：Q11(8次)、Q14(5次)、Q19(4次) 等
**根因**：虽然 prompt 已加效率规则，但 agent 仍倾向"多角度验证"。平均 3.7 次/题，理想值应为 2-3 次。
**严重程度**：低 — 功能正确但效率有优化空间。

### 改进：Q38 未调用工具直接回答

**影响范围**：Q38（CVPR 2026 不存在）
**根因**：agent 知道 CVPR 不在数据库覆盖范围内，直接回答而不查询。评测脚本判定为路由错误。
**严重程度**：低 — 这是评测标准问题而非 agent 问题。可考虑调整 Q38 的评测逻辑。

## 五、优化建议

### 建议 1（重要）：扩展 mixed 题的工具选择触发词

**影响**：Q23、Q24、Q34 路由失败
**预期**：mixed 路由正确率从 76.9% 提升到 ~90%

```diff
--- a/src/deep_paper_qa/prompts.py
+++ b/src/deep_paper_qa/prompts.py
@@ -124,3 +124,5 @@
 工具选择规则：
-- 统计/计数/排名/趋势问题 → execute_sql（用全文检索做主题过滤）
+- 统计/计数/排名/趋势问题 → execute_sql（用全文检索做主题过滤）
+- 涉及"哪个会议/哪一年/高引/按会议/按年份"等元数据维度的问题，即使主要是内容查询，也应先用 execute_sql 定位论文范围
+- 混合问题判断：如果问题同时涉及"某个领域的论文"和"具体内容/方法/效果"，一律按混合处理
 - 精确关键词/术语搜索 → search_abstracts（返回论文列表+摘要片段）
```

### 建议 2（改进）：强化效率规则的约束力

**影响**：Q11 等 content 题调用次数过多
**预期**：content 题平均调用从 3.7 降到 2-3 次

```diff
--- a/src/deep_paper_qa/prompts.py
+++ b/src/deep_paper_qa/prompts.py
@@ -132,5 +132,5 @@
 效率规则：
-- 优先用一次工具调用获取足够信息，不要反复用不同关键词搜索同一主题
-- 如果第一次搜索已返回 5+ 条相关结果，直接基于这些结果回答，不要再搜
-- 对同一问题，不要同时用 search_abstracts 和 vector_search 搜索相似内容，选一个即可
-- 混合问题最多 2-3 次工具调用：先统计再内容，不要反复验证
+- 严格控制：内容类问题最多调用 2 次搜索工具（search_abstracts 或 vector_search 二选一，不要两个都用）
+- 如果第一次搜索已返回 3+ 条相关结果（相关度 > 0.05），立即停止搜索，基于已有结果回答
+- 混合问题最多 3 次工具调用：1 次 SQL + 1-2 次搜索
+- 绝对上限：任何单个问题不超过 4 次工具调用
```
