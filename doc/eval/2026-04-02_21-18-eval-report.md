## 四、问题发现

### 重要：mixed 题仍有 20% 缺少 execute_sql

**影响范围**: Q22、Q24、Q33（3/15 = 20%）
**表现**: Agent 使用 search_abstracts 的 where 参数替代 execute_sql 做筛选，跳过统计步骤
**典型案例**:
- Q22 "ACL 2024 关于 CoT 的论文实验结果"：Agent 用 search_abstracts(where="year=2024 AND conference='ACL'") 替代 execute_sql
- Q33 "AAAI 2024 关于数据增强的论文"：同上
**根因**: search_abstracts 的 where 参数能实现部分 execute_sql 的筛选功能，Agent 合理地选择了更简洁的路径。这不一定是 bug，但评测逻辑要求 mixed 题必须用 execute_sql。

### 重要：冗余调用仍然存在

**影响范围**: 10 道题 >4 次调用，最高 Q55 达 9 次
**高发区域**: content 题（平均 3.4 次/题）和 mixed 题（平均 3.9 次/题）
**模式**: 同一工具连续调用 2-3 次，每次换不同关键词
**量化**: 相比上次报告（40 题平均 3.0 次），本次 55 题平均 2.7 次/题，但 content 题均值未改善

### 改进：Q38 和 Q53 评测逻辑

**Q38** "CVPR 2026"：Agent 知道数据库不含 CVPR，直接回答，行为合理但被判错
**Q53** "asdfghjkl"：Agent 识别为无意义输入直接回答，行为合理但被判错
**根因**: 评测逻辑要求 sql 题必须调 execute_sql、content 题必须调搜索工具，但对"不需要调工具"的场景没有容错

### 观察：线上日志与评测一致

线上 16 次真实用户查询中：
- execute_sql 被调用 43 次（统计类查询主导）
- search_abstracts 被调用 21 次
- vector_search 仅被调用 5 次（线上用户更偏好精确搜索）
- 评测中 vector_search 测试充足（13 题），线上低使用率可能因用户习惯

## 五、优化建议

### 建议 1：评测逻辑容错——允许合理的不调用

影响：Q38、Q53（2 题）
预期：sql 正确率从 93.8% 回 100%，content 正确率维持/提升

```diff
--- a/eval/run_eval.py
+++ b/eval/run_eval.py
@@ -63,7 +63,13 @@
     # 评估工具路由正确性
     tool_correct = False
     called_set = set(tools_called)
     if q_type == "sql":
-        tool_correct = "execute_sql" in called_set
+        # Agent 直接回答（0 次调用）且回答合理也视为正确
+        tool_correct = "execute_sql" in called_set or (
+            len(called_set) == 0 and any(
+                kw in final_answer for kw in ["不存在", "没有", "不包含", "无法", "不在"]
+            )
+        )
     elif q_type == "content":
-        tool_correct = bool(called_set & {"vector_search", "search_abstracts"})
+        # 无意义输入直接回答也视为正确
+        tool_correct = bool(called_set & {"vector_search", "search_abstracts"}) or (
+            len(called_set) == 0 and any(
+                kw in final_answer for kw in ["无法理解", "请提供", "没有找到", "无意义"]
+            )
+        )
```

### 建议 2：prompt 中强化效率硬限制

影响：10 道题 >4 次调用（Q11 7次、Q55 9次）
预期：content 平均调用从 3.4 降到 2.5，mixed 从 3.9 降到 3.0

```diff
--- a/src/deep_paper_qa/prompts.py
+++ b/src/deep_paper_qa/prompts.py
@@ -132,6 +132,8 @@
 效率规则：
+- **硬限制：单个问题最多调用 4 次工具**。超过 4 次必须立即基于已有信息回答。
+- 禁止连续用同一工具搜索 2 次以上（换关键词也算）
 - 优先用一次工具调用获取足够信息，不要反复用不同关键词搜索同一主题
 - 如果第一次搜索已返回 5+ 条相关结果，直接基于这些结果回答，不要再搜
 - 对同一问题，不要同时用 search_abstracts 和 vector_search 搜索相似内容，选一个即可
```

### 建议 3：强化 mixed 题路由——先统计再内容

影响：Q22、Q24、Q33（3 题）
预期：mixed 路由正确率从 80% 提升到 90%+

```diff
--- a/src/deep_paper_qa/prompts.py
+++ b/src/deep_paper_qa/prompts.py
@@ -128,1 +128,2 @@
-- 混合问题 → 先 execute_sql 做统计，再 search_abstracts 或 vector_search 查内容
+- 混合问题（同时涉及统计筛选和内容理解）→ **必须**先用 execute_sql 做统计/筛选，再用检索工具查内容
+  即使 search_abstracts 的 where 参数能实现筛选，仍应优先用 execute_sql，因为它返回精确计数和排序
```

### 建议 4：优化 content 题路由——避免不必要的 execute_sql

影响：Q12、Q13、Q15、Q18、Q37、Q39 等 content 题多调了 execute_sql
预期：content 题平均调用数从 3.4 降到 2.0

```diff
--- a/src/deep_paper_qa/prompts.py
+++ b/src/deep_paper_qa/prompts.py
@@ -126,1 +126,2 @@
 - 模糊/概念性搜索 → vector_search（基于语义相似度）
+- 纯内容理解问题（"XX 是什么"/"XX 怎么工作"）→ 只用 vector_search 或 search_abstracts，不要额外调 execute_sql 做统计
```
