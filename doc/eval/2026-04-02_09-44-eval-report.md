## 四、问题发现

### 致命：recursion_limit=10 过低

**影响范围：** 80% 的题目（32/40）
**根因：** LangGraph ReAct agent 每次工具调用消耗约 3 步（LLM 思考 → tool_call → tool_result），recursion_limit=10 仅允许约 3 次工具调用。而 agent 实际平均需要 3-5 次调用。
**线上验证：** 线上日志显示平均 4.7 次工具调用/回答，agent 配置中无 recursion_limit 限制（使用默认值 25）。

### 重要：content 类路由评估标准偏差

**影响范围：** Q31-32 等 content 题
**根因：** `questions.jsonl` 中 content 题的 `expected_tool` 全是 `"vector_search"`，但实际上 agent 对关键词明确的问题更倾向用 `search_abstracts`。评测脚本只检查 `"vector_search" in tools_called`，导致 agent 用 search_abstracts 时也算"正确"（因为最终也会调 vector_search）。Q31-32 的 expected_tool 设为 `"search_abstracts"` 但评测脚本 content 分支硬编码检查 vector_search。

### 改进：Agent 多角度验证导致冗余

**影响范围：** 几乎所有 content 和 mixed 题
**表现：** agent 习惯性地用 search_abstracts 搜 2 次（不同关键词）+ vector_search 搜 1 次，即使第一次搜索已返回足够结果。这在线上场景（recursion_limit=25）下只是慢，但在评测（limit=10）下直接导致超限。

### 改进：run_eval.py 未记录耗时和详细 tool 输入输出

**影响范围：** 评测分析精度
**根因：** 当前 `eval_results.json` 只记录 `tools_called`（名称列表）和 `answer_preview`，缺少每次调用的输入参数、输出内容和耗时，无法做深入的效率分析。
