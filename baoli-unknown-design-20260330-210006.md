## Problem Statement

科研人员在面对海量 AI 论文时，缺少一个能用自然语言对话回答多层次问题的助手。现有工具（如 PaperQA）只能回答论文内容级问题（"这篇论文说了什么"），无法回答元数据/统计级问题（"ACL 2025 发了多少篇论文"、"2025 年高引 RAG 论文有哪些"）。用户已有混合检索 API（语义 + 结构化查询），需要在此基础上构建智能问答层。

## What Makes This Cool

**PaperQA 做不到的事情，你能做到。** 你问它"EMNLP 2025 有多少篇关于 Agent 的论文"，它直接给你数字和分布，而不是甩一堆摘要让你自己数。这是和所有现有论文 QA 系统的关键差异。

核心 delight：
- 自然语言提问，系统自动判断是内容检索还是统计查询还是混合
- 多轮对话追问，像和一个懂行的同事聊天
- 回答附带引用来源，统计类结果用结构化格式展示
- ReAct 模式让推理过程可观察 —— 用户能看到 Agent "在想什么"

## Constraints

- 检索能力依赖已有的混合检索 API（不在本项目范围内）
- 技术栈偏好：LangGraph 1.0 + ReAct 模式、LiteLLM、Instructor、Pydantic、loguru、python-dotenv
- Python 环境使用 uv 管理
- 模型/数据源优先 ModelScope
- 面向开源社区，代码需要易读、易贡献
- 异步 I/O 必须用 asyncio

## Premises

1. **检索 API 已支持结构化元数据查询**（会议、年份、引用数、方向等），不仅是文本语义检索
2. **核心体验是自然语言对话**，系统自动判断问题类型并路由到对应工具
3. **核心价值不是检索，而是"理解问题 + 组织答案"** —— 检索是 API 的事，本项目做 Agent 层

## Cross-Model Perspective

独立 Claude subagent 审查了项目方向，关键洞察：

- **最酷版本：** "研究领域的 Bloomberg Terminal" —— 不是你问它才答，而是主动推送研究态势变化（10x 版本，非 MVP）
- **核心兴奋点：** "像一个知识渊博的同事" —— 要的不是工具，是关系。重点应投入对话深度和推理可解释性
- **50% 的路：** LangGraph 提供 Agent 编排、状态管理、工具调用；需自建 Query Router、结构化查询生成器、答案综合引擎、学术 Prompt 工程
- **周末 MVP：** Query Router + 双通道调用 demo，唯一目标让人 30 秒内理解"和 PaperQA 有什么不同"

## Approaches Considered

### Approach A: 轻量级 Agent（LiteLLM + Instructor）
纯 Python 实现 Agent 循环，不用框架。Query Router 用 Instructor 做结构化输出。
- 工作量：S
- 风险：低
- 优势：依赖少、符合技术栈偏好、容易理解和贡献
- 劣势：多轮对话状态管理需自己写，扩展性有限

### Approach B: LangGraph 1.0 + ReAct（用户选择）
基于 LangGraph 1.0 的 ReAct Agent，类 Claude Code 的模式。Agent 有一组 tools（content_search、metadata_query、paper_detail 等），通过 ReAct 循环自主决定调用哪个工具、如何组织答案。
- 工作量：M
- 风险：中
- 优势：多轮对话天然支持、ReAct 推理过程可观察、复杂问题自动分解、社区生态活跃
- 劣势：依赖较重、LangGraph 学习曲线

### Approach C: 多 Agent 协作
拆分为检索 Agent、统计 Agent、综述 Agent，由 Orchestrator 协调。
- 工作量：L
- 风险：高
- 优势：最接近"全能体"愿景
- 劣势：过度设计风险、初期无必要

## Recommended Approach

**Approach B: LangGraph 1.0 + ReAct**

用户明确选择。理由充分 —— ReAct 模式让 Agent 自主决定工具调用顺序，天然适合"有时查内容、有时查统计、有时两者都要"的场景。LangGraph 的状态管理解决多轮对话问题，checkpointer 支持对话持久化。

### 核心架构

```
用户输入 (自然语言)
    ↓
LangGraph ReAct Agent (LLM + System Prompt)
    ↓ (思考 → 选择工具 → 执行 → 观察 → 循环)
    ├── content_search(query, filters) → 检索 API (语义检索)
    ├── metadata_query(conference, year, direction, sort_by, ...) → 检索 API (结构化查询)
    ├── paper_detail(paper_id) → 获取论文详情
    └── aggregate_stats(field, group_by, ...) → 统计聚合
    # Phase 2: citation_analysis(paper_ids) → 引用分析
    ↓
最终回答 (带引用/表格/结构化输出)
```

### 关键组件

1. **Tools 定义（Pydantic schemas）**

   检索 API 以 HTTP 服务形式提供（如 FastAPI），tools 通过 aiohttp 异步调用。

   ```python
   # content_search: 语义检索，基于用户问题检索相关论文段落
   # 典型调用: content_search(query="RAG chunking 策略改进", top_k=10)
   class QueryFilters(BaseModel):
       year: int | None = None
       conference: str | None = None
       direction: str | None = None
       authors: list[str] | None = None
       min_citations: int | None = None

   class ContentSearchInput(BaseModel):
       query: str          # 语义检索 query
       top_k: int = 10     # 返回数量
       filters: QueryFilters | None = None

   class PaperMetadata(BaseModel):
       year: int
       conference: str | None = None
       authors: list[str]
       citations: int
       direction: list[str]       # 研究方向标签
       abstract: str | None = None

   class ChunkResult(BaseModel):
       paper_id: str
       title: str
       chunk_text: str
       score: float
       metadata: PaperMetadata

   # metadata_query: 结构化条件查询，返回符合条件的论文列表
   # 典型调用: metadata_query(conference="ACL", year=2025, direction="RAG", sort_by="citations", limit=20)
   # 与 aggregate_stats 的区别: metadata_query 返回论文列表，aggregate_stats 返回统计数字
   class MetadataQueryInput(BaseModel):
       conference: str | None = None
       year: int | None = None
       direction: str | None = None
       sort_by: str = "relevance"  # relevance | citations | date
       limit: int = 20

   # paper_detail: 获取单篇论文完整信息
   # 典型调用: paper_detail(paper_id="arxiv:2312.07559")
   class PaperDetailInput(BaseModel):
       paper_id: str

   # citation_analysis: [Phase 2 功能，MVP 不实现] 分析论文间引用关系
   # 典型调用: citation_analysis(paper_ids=["id1", "id2"], analysis_type="common_references")
   # class CitationAnalysisInput(BaseModel):
   #     paper_ids: list[str]
   #     analysis_type: str = "common_references"

   # aggregate_stats: 统计聚合，返回数字/分布而非论文列表
   # 典型调用: aggregate_stats(field="conference", group_by="year", filters={"direction": "RAG"})
   # 回答类似"ACL 2025 发了多少篇"这种问题
   class AggregateStatsInput(BaseModel):
       field: str           # 聚合字段: conference | direction | year
       group_by: str | None = None
       aggregation: str = "count"  # count | avg | distribution
       filters: QueryFilters | None = None
   ```

2. **System Prompt（学术领域专家人设）**
   - 引导 Agent 判断问题类型
   - 规范引用格式
   - 定义回答风格（简洁、有依据、附引用）

3. **State Management（LangGraph Checkpointer）**
   - 多轮对话上下文持久化
   - 支持 memory-saver（开发）和 postgres-saver（生产）

4. **前端（Gradio / Chainlit）**
   - 聊天界面，支持多轮对话
   - 统计结果结构化展示（表格/列表）
   - 推理过程可观察（展示 Agent 的思考步骤）
   - 注：Gradio Chatbot 对流式中间步骤支持有限，Chainlit 对 Agent 对话展示更友好，实现时需验证并选择

### 技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| Agent 框架 | LangGraph 1.0 (langgraph + langgraph-checkpoint) | ReAct Agent 核心 |
| LLM 调用 | LiteLLM | 通过 OpenAI 兼容接口对接 LangGraph（`ChatLiteLLM` 或自定义 `BaseChatModel` 适配） |
| 结构化输出 | Pydantic v2 | Tool schema 定义用 Pydantic；Instructor 仅在需要从 LLM 自由文本中提取结构化数据时使用（如解析用户意图为 enum） |
| 前端 | Gradio / Chainlit（待验证） | 需验证 Agent 中间步骤流式展示能力 |
| 日志 | loguru | — |
| 配置 | python-dotenv | — |
| 包管理 | uv | — |
| 异步 | asyncio + aiohttp | 所有 API 调用异步 |

### 错误处理与降级策略

- **检索 API 不可用**: 返回友好提示"检索服务暂不可用，请稍后再试"，不让 Agent 幻觉编造答案
- **LLM 输出校验**: Tool 调用参数通过 Pydantic 校验，不合法则要求 LLM 重试（LangGraph 内置 retry 机制）
- **Token 超限**: 检索结果过多时截断，保留 top-k 最相关 chunks，在 system prompt 中限制最大上下文长度
- **超时**: 单次 tool 调用设 30s 超时，超时后 Agent 跳过该 tool 并告知用户
- **幻觉防护**: 回答必须基于 tool 返回结果，system prompt 明确禁止编造论文信息
- **空结果处理**: API 正常但查无结果时，Agent 告知用户"未找到匹配结果"并建议放宽查询条件，不编造答案

## Success Criteria

- 准备 30 个测试问题（10 内容级 / 10 统计级 / 10 混合），准确率 >= 80%
- 内容级问题回答必须附带论文引用（title + paper_id）
- 统计级问题回答必须给出准确数字（与检索 API 直查结果一致）
- 多轮对话能保持上下文（追问"其中哪些是关于 agentic RAG 的"能正确理解指代）
- 内容级/混合问题单轮响应时间 < 15s；统计聚合类取决于检索 API 性能，MVP 不硬性约束
- Agent 推理过程对用户可见（至少展示 tool 调用和返回摘要）
- README 清晰，新贡献者 10 分钟内能跑起来

## What I noticed about how you think

- 你说"检索我已经在另一个项目做了，可以作为 API 使用" —— 你习惯把系统拆成独立模块，每个模块有清晰的边界。这是工程素养的体现。
- 你选了"科研助手全能体"但立刻把范围收到"现在关注问答" —— 野心大但执行务实，知道什么时候该聚焦。
- 你主动提出"基于 LangGraph 1.0，用 ReAct 作为基础，也就是 Claude Code 的模式" —— 你对 Agent 架构有自己的判断，不是被动接受建议。
- 面向开源社区而不是自己用 —— 你在意代码质量和项目的可参与性，不只是解决自己的问题。
